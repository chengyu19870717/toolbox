# ToolBox 部署运维指南

> 版本：v1.0
> 文档定位：从 Mac 上构建 Windows 绿色版到内网服务器部署运行的完整流程，包含 Windows 服务化、备份升级、故障排查、Windows 部署冒烟测试清单。
> 受众：你（构建者 + 运维），未来可能给团队的运维同学。

---

## 1. 部署形态总览

```
开发环境 (Mac)                              内网环境 (Windows Server)
┌───────────────────────────┐              ┌───────────────────────────┐
│ macOS                     │              │ Windows Server / 10 / 11  │
│ ├── JDK 21 (Mac)          │              │                           │
│ ├── Node.js 20+           │   构建打包    │ ToolBox/                  │
│ ├── 主项目源码             │ ──────────→  │  ├── jre/ (Win JRE 21)    │
│ └── Parallels             │   ToolBox.zip │  ├── lib/                 │
│     └── Windows 11 (测试) │              │  ├── plugins/             │
└───────────────────────────┘              │  └── ...                  │
                                           │  以 Windows 服务运行       │
                                           │  9090 端口对内网开放       │
                                           └───────────────────────────┘
```

**核心特性**：
- **绿色版**：解压即用，不需要在目标机器装 JDK/JRE
- **可服务化**：可通过 WinSW 注册为 Windows 服务，开机自启
- **可升级**：替换 jar 即可升级，配置和数据保留
- **零网络依赖**：部署时不需要联网（依赖都打进绿色版了）

## 2. 构建打包（Mac 侧）

### 2.1 前提

```bash
# JDK 21
java -version
# openjdk version "21.x.x"

# Node.js
node -v   # v20+
npm -v    # 10+

# 主项目根目录
cd ~/projects/toolbox
```

### 2.2 一键打包

```bash
./gradlew clean packageWindows
```

执行内容：
1. 清理 `build/` 和 `dist/`
2. 安装前端依赖（pnpm/npm install）
3. 构建主程序前端（`frontend/dist/`）
4. 把前端产物嵌入 `framework/src/main/resources/static/`
5. 构建主程序 fat jar（`framework/build/libs/toolbox-app.jar`）
6. 构建所有示例插件 jar（`plugin-*/build/libs/`）
7. 下载 Windows JRE 21（首次执行从 Adoptium 拉，缓存到 `~/.gradle/jre-cache/`）
8. 组装绿色版目录结构
9. 打包成 zip：`dist/ToolBox-{version}-windows.zip`

**首次执行**会下载 ~80MB 的 Windows JRE，后续构建直接用缓存（<2 分钟）。

### 2.3 自定义打包参数

```bash
# 指定版本号
./gradlew packageWindows -Pversion=1.0.0-rc1

# 不包含示例插件
./gradlew packageWindows -PincludeSamples=false

# JRE 版本（默认 21）
./gradlew packageWindows -PjreVersion=21.0.5+11
```

### 2.4 验证产物

```bash
ls -lh dist/
# ToolBox-1.0.0-windows.zip   (~150MB)

# 解压检查（在 Mac 上展开看一下结构）
unzip -l dist/ToolBox-1.0.0-windows.zip | head -30
```

期望结构：

```
ToolBox/
├── bin/
│   ├── start.bat
│   ├── stop.bat
│   ├── install-service.bat
│   ├── uninstall-service.bat
│   ├── toolbox.vmoptions
│   ├── winsw.exe
│   └── winsw.xml
├── jre/
│   └── ... (Windows JRE 21)
├── lib/
│   └── toolbox-app.jar
├── plugins/
│   ├── plugin-sample-sql-runner-1.0.0.jar
│   ├── plugin-sample-excel-importer-1.0.0.jar
│   └── plugin-sample-flow-viewer-1.0.0.jar
├── config/
│   └── application.yml          (默认配置模板)
├── README.txt
└── VERSION
```

## 3. 在 Parallels 虚拟机里冒烟测试

### 3.1 准备 Windows 虚拟机

- Windows 10 / 11 都行
- 至少 4 GB 内存（推荐 8 GB）
- 至少 5 GB 可用磁盘
- 装一个 Chrome 或 Edge

### 3.2 把 zip 拷进去

Mac → Windows VM 几种方法：
- Parallels 共享文件夹（最方便）
- 拖拽到 VM 窗口
- 通过 OneDrive / Google Drive 中转
- 通过本地 HTTP 服务（`python3 -m http.server`，VM 浏览器下载）

### 3.3 解压

**关键**：解压目录要避开**深路径**和**带空格的路径**：

```
✅ D:\toolbox\
✅ C:\apps\toolbox\

❌ C:\Users\张三\Documents\我的项目 (xx)\toolbox\   (中文 + 空格 + 深路径)
```

### 3.4 首次启动

```cmd
cd D:\toolbox\bin
```

## 5. 内网部署

冒烟测试通过后，把 zip 传到内网服务器。

### 5.1 服务器要求

| 项 | 推荐 | 最低 |
|---|---|---|
| 操作系统 | Windows Server 2019/2022 | Windows 10/11 |
| CPU | 4 核 | 2 核 |
| 内存 | 8 GB | 4 GB |
| 磁盘 | 20 GB（数据增长按需扩） | 5 GB |
| 网络 | 内网静态 IP | 同左 |

### 5.2 传输

通过你的内网传输工具把 `ToolBox-1.0.0-windows.zip` 传到服务器，比如 `D:\` 下。

### 5.3 解压与首次启动

```cmd
cd D:\
:: 用 Windows 内置的 PowerShell 解压（或用 7zip）
powershell -Command "Expand-Archive -Path ToolBox-1.0.0-windows.zip -DestinationPath ."

cd D:\ToolBox\bin

:: 编辑 application.yml 修改端口（如果 9090 被占用）
notepad ..\config\application.yml

:: 第一次先用前台启动看看
start.bat
```

如果一切正常，浏览器访问 `http://<服务器IP>:9090`。

### 5.4 注册为 Windows 服务

确认前台启动 OK 后，停掉，注册为服务：

```cmd
:: 必须以管理员身份运行 cmd
cd D:\ToolBox\bin

install-service.bat
```

注册后：

```cmd
:: 启动
sc start ToolBox

:: 或者 services.msc 里手动启动

:: 状态查询
sc query ToolBox
```

### 5.5 防火墙

让团队能访问 9090 端口：

```cmd
:: 允许 9090 端口入站
netsh advfirewall firewall add rule name="ToolBox" dir=in action=allow protocol=TCP localport=9090
```

或者通过"控制面板 → 防火墙 → 高级设置"图形化添加。

### 5.6 修改默认账号

部署后**第一时间**：

1. 浏览器登录 admin / admin123
2. 强制改密（设置一个强密码）
3. 创建团队成员账号
4. 妥善保管 admin 密码

### 5.7 配置数据源

1. 数据源管理 → 新增
2. 填入测试库连接信息
3. 测试连接通过
4. 通知团队成员可以使用

## 6. 配置详解

### 6.1 application.yml

```yaml
server:
  port: 9090                       # 修改这里换端口
  servlet:
    context-path: /                # 一般不改

toolbox:
  paths:
    config-dir: config
    plugins-dir: plugins
    plugins-data-dir: plugins-data
    temp-dir: temp
    logs-dir: logs

  security:
    # JWT 密钥 - 生产部署必须修改！否则别人能伪造 token
    jwt-secret: REPLACE_ME_IN_PRODUCTION_AT_LEAST_32_CHARS
    jwt-expire-hours: 8

  task:
    max-concurrent-per-user: 5     # 每用户并发任务数上限
    retention-days: 30             # 任务结果保留天数
    cleanup-cron: "0 0 3 * * *"    # 每天凌晨 3 点清理

  file:
    upload-max-size: 100MB         # 上传大小上限
    temp-retention-hours: 24       # 临时文件保留时间

  datasource:
    default-query-timeout-seconds: 60      # 默认查询超时
    default-max-rows: 1000000              # 默认结果集行数上限

logging:
  level:
    root: INFO
    com.toolbox: INFO
    # 调试某个插件时改这里
    # com.toolbox.plugin.sqlrunner: DEBUG
  file:
    name: logs/app.log
  logback:
    rollingpolicy:
      max-history: 30              # 保留 30 天
      max-file-size: 50MB          # 单文件最大
      total-size-cap: 5GB          # 所有日志文件总大小上限
```

### 6.2 toolbox.vmoptions（JVM 参数）

```
# 堆内存（默认 -Xms256m -Xmx2g）
-Xms256m
-Xmx2g

# 编码
-Dfile.encoding=UTF-8
-Dsun.jnu.encoding=UTF-8

# 时区
-Duser.timezone=Asia/Shanghai

# 堆 dump（OOM 时）
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=logs/heap-dump.hprof

# GC 日志（可选，调优时打开）
# -Xlog:gc*:file=logs/gc.log:time:filecount=10,filesize=10M
```

**调整建议**：
- 服务器内存充足时，把 `-Xmx` 调大（比如 `-Xmx4g`）
- 内存吃紧时调小，但不低于 `-Xmx512m`

### 6.3 winsw.xml（Windows 服务配置）

```xml
<service>
  <id>ToolBox</id>
  <name>ToolBox</name>
  <description>ToolBox - 内部工具集</description>

  <!-- 工作目录 -->
  <workingdirectory>%BASE%\..</workingdirectory>

  <!-- 启动命令 -->
  <executable>%BASE%\..\jre\bin\java.exe</executable>
  <arguments>@%BASE%\toolbox.vmoptions -jar %BASE%\..\lib\toolbox-app.jar --spring.config.location=file:%BASE%\..\config\application.yml</arguments>

  <!-- 日志 -->
  <logpath>%BASE%\..\logs</logpath>
  <log mode="roll-by-size">
    <sizeThreshold>10240</sizeThreshold>
    <keepFiles>10</keepFiles>
  </log>

  <!-- 启动模式 -->
  <onfailure action="restart" delay="10 sec"/>
  <onfailure action="restart" delay="30 sec"/>
  <onfailure action="none"/>

  <!-- 优雅停机超时 -->
  <stoptimeout>30 sec</stoptimeout>
</service>
```

通常**不需要改**这个文件，除非要调整服务名或日志路径。

### 6.4 数据库 JDBC URL 参数

数据源的 `params` 字段是 JDBC URL 参数。常用：

```json
{
  "useUnicode": "true",
  "characterEncoding": "utf8mb4",
  "useSSL": "false",
  "allowPublicKeyRetrieval": "true",
  "serverTimezone": "Asia/Shanghai",
  "connectTimeout": "10000",
  "socketTimeout": "60000"
}
```

**OceanBase MySQL 模式**：基本参数和 MySQL 一样，建议加：

```json
{
  "useSSL": "false",
  "useServerPrepStmts": "false",
  "useUnicode": "true",
  "characterEncoding": "utf8mb4",
  "serverTimezone": "Asia/Shanghai"
}
```

## 7. 升级流程

### 7.1 单插件升级（最常用）

```cmd
:: 1. 准备新版本 jar，比如 plugin-sql-runner-1.1.0.jar

:: 2. 拷贝到 plugins/ 目录
copy plugin-sql-runner-1.1.0.jar D:\ToolBox\plugins\

:: 3. 删除旧版本
del D:\ToolBox\plugins\plugin-sql-runner-1.0.0.jar

:: 4. 重启服务
sc stop ToolBox
sc start ToolBox

:: 5. 浏览器验证新版本生效
```

**回滚**：把旧版本 jar 放回去，删新版本，重启。

### 7.2 框架升级

```cmd
:: 1. 备份当前部署
xcopy D:\ToolBox D:\ToolBox-backup-2026-05-09 /E /I

:: 2. 停服务
sc stop ToolBox

:: 3. 替换 lib/toolbox-app.jar
copy new-toolbox-app.jar D:\ToolBox\lib\toolbox-app.jar

:: 4. 启动服务
sc start ToolBox

:: 5. 验证
```

### 7.3 整体替换升级

适用于框架 + 多个插件同时升级：

```cmd
:: 1. 解压新版本到旁边
cd D:\
powershell -Command "Expand-Archive -Path ToolBox-2.0.0-windows.zip -DestinationPath .\ToolBox-new"
```

## 10. 故障排查

### 10.1 服务起不来

**症状**：`sc start ToolBox` 失败 / `start.bat` 闪退

**排查步骤**：

1. **看日志**：
   ```
   D:\ToolBox\logs\app.log
   D:\ToolBox\logs\winsw.out.log
   D:\ToolBox\logs\winsw.err.log
   ```

2. **常见原因**：

   | 错误 | 解决 |
   |---|---|
   | 端口被占用 | `netstat -ano \| findstr :9090` 找占用进程；或改 `application.yml` 端口 |
   | JRE 损坏 / 缺文件 | 解压时被杀毒软件拦截，加白名单后重新解压 |
   | 路径含中文/空格 | 移到 `D:\toolbox\` 类纯英文路径 |
   | application.yml 格式错误 | YAML 缩进错误，对照默认模板 |
   | 配置文件权限不足 | 给 ToolBox 目录读写权限 |

3. **手动启动看完整报错**：
   ```cmd
   cd D:\ToolBox\bin
   start.bat
   ```
   命令行窗口能看到完整错误堆栈。

### 10.2 浏览器打不开页面

**排查步骤**：

1. 服务进程是否在跑：`sc query ToolBox` → STATE 应为 RUNNING
2. 端口是否监听：`netstat -ano | findstr :9090`
3. 防火墙：在服务器本机用 `http://localhost:9090` 能访问？能 → 防火墙问题
4. 内网网络：`ping <服务器IP>` 通不通

### 10.3 登录页打不开 / 一直白屏

- F12 看浏览器控制台报错
- 常见：JS 加载失败 → 是不是路径含中文导致静态资源 404
- Network 面板看 `index.html` 和 `*.js` 文件返回什么

### 10.4 数据源连不上

**症状**：测试连接报错

**排查**：

1. 服务器 ping 数据库主机通不通
2. telnet 端口通不通：`telnet <db-host> 3306`
3. 数据库账号是不是允许从服务器 IP 登录（`mysql.user` 表的 host 字段）
4. 数据库本身是否启动
5. SSL 配置：URL 加 `useSSL=false`
6. 时区：URL 加 `serverTimezone=Asia/Shanghai`

### 10.5 SQL 跑得慢 / 超时

- 检查 SQL 本身（`EXPLAIN`）
- 数据库压力（其他人也在用这个库？）
- 加索引
- 修改数据源的"高级配置"调大超时
- 单次返回行数过大 → 减少 `maxRows` 或加 LIMIT

### 10.6 任务一直 PENDING 不执行

- 当前用户是否已经有 5 个 RUNNING 任务？看任务列表
- 任务线程池是否阻塞？看 `logs/app.log` 是否有 ERROR
- 重启服务（最后手段）

### 10.7 任务取消不响应

**原因**：插件没正确响应 `Thread.interrupt()`

- 看 `logs/plugin-{id}.log`，看任务是否真的被中断
- 联系插件作者修复
- 临时方案：重启服务

### 10.8 文件上传失败

| 错误 | 解决 |
|---|---|
| 413 Payload Too Large | 文件超 100MB，调整 application.yml 的 upload-max-size |
| 上传中断 | 网络问题，重试；或调整 server.tomcat.connection-timeout |
| 上传后处理失败 | 文件格式问题，看 logs/plugin-*.log |

### 10.9 内存不足 / OOM

**症状**：`logs/app.log` 出现 `OutOfMemoryError` / 服务自动重启

**解决**：

1. 看 `logs/heap-dump.hprof`（用 MAT 分析）找内存泄漏
2. 调大 `toolbox.vmoptions` 里的 `-Xmx`
3. 减小数据源的 `maxRows`（避免一次返回太多行）
4. 临时方案：重启服务

### 10.10 磁盘空间不足

**容易吃磁盘的目录**：

```
logs/                    日志文件（按天滚动 + 总量上限 5GB）
temp/                    上传下载临时文件（24h 自动清）
data/tasks.db            SQLite（30 天清理）
plugins-data/            插件数据（不主动清，关注插件作者实现）
```

**应急清理**：

```cmd
:: 清理超过 7 天的日志
forfiles /P D:\ToolBox\logs /M *.log /D -7 /C "cmd /c del @path"

:: 清理临时文件
del /Q D:\ToolBox\temp\uploads\*
del /Q D:\ToolBox\temp\downloads\*
```

### 10.11 收集信息提交问题

如果排查不出，需要把以下信息打包给开发者：

```
- ToolBox 版本（VERSION 文件）
- Windows 版本
- application.yml
- logs/app.log (最后 1000 行)
- logs/plugin-*.log (有问题的插件)
- 复现步骤
- 浏览器 F12 截图
```

## 11. 安全建议

虽然是内网，仍建议：

| 项 | 建议 |
|---|---|
| 默认密码 | 部署后立即改 admin 密码 |
| JWT 密钥 | 部署时改 `application.yml` 的 jwt-secret，至少 32 字符随机串 |
| 数据库账号 | 给 ToolBox 用的数据库账号只给查询权限（除非工具确实要写） |
| 服务运行账号 | Windows 服务别用 LocalSystem，建专门的服务账号 |
| 日志清理 | 日志可能有敏感信息（用户名等），定期清理 |
| 备份保护 | 备份目录权限设严格，不要全员可读 |

**已知不做的安全特性**（按之前需求确认）：
- 不加密数据源密码（明文存 JSON）
- 不做 HTTPS（内网 HTTP）
- 不做立即踢下线
- 不做请求审计日志

如未来要做，再谈。

## 12. 性能调优

### 12.1 JVM 参数

适合中小负载（<20 用户）的默认配置：

```
-Xms512m
-Xmx2g
```

适合较高负载（20-50 用户）：

```
-Xms1g
-Xmx4g
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
```

### 12.2 数据库连接池

每个数据源的 `poolConfig` 默认：

```json
{
  "maximumPoolSize": 10,
  "minimumIdle": 2,
  "connectionTimeoutMs": 30000
}
```

调整建议：
- 用户多 → maximumPoolSize 调到 20-30
- 长查询多 → connectionTimeoutMs 调大
- 数据库连接数受限 → maximumPoolSize 调小，避免占满

### 12.3 任务并发数

```yaml
toolbox:
  task:
    max-concurrent-per-user: 5     # 默认 5
```

增加并发：CPU/内存吃得消的话，调到 10
减少并发：用户多 + 任务重，可以调到 3

## 15. 工具菜单：FAQ

**Q: 多人能不能同时开同一个工具？**
A: 能。每人的 Tab 互相独立。

**Q: 关掉浏览器，正在跑的任务怎么样？**
A: 任务在后端继续跑（与浏览器无关），下次登录可以在任务列表里看到状态和结果。

**Q: 服务器重启后任务怎么样？**
A: 没跑完的任务全部 CANCELLED，按设计不恢复。已完成的任务和产物都在。

**Q: 改 application.yml 后要重启吗？**
A: 要。Spring Boot 不支持配置热加载（用了 Spring Cloud Config 才行，咱们没用）。

**Q: 一个插件能装两个版本吗？**
A: 不能。同 plugin.id 多版本启动会报错。

**Q: 升级到下一个 LTS Java 版本怎么搞？**
A: 重新打包绿色版即可（`packageWindows -PjreVersion=...`），框架代码本身一般不需要改。

**Q: 团队规模大了能跑得起来吗？**
A: 第一版的容量目标：<50 用户、<200 数据源、<1000 个并发任务。再大需要做集群化（多实例 + 负载均衡），那是后续版本的事。
